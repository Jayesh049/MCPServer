Where does the data come from?

This platform is a medical education demo. All data sources are clearly labeled throughout the interface so you always know what you're seeing and why.

Data Sources
📖
Wikipedia — Medical Articles
Patient chat answers are grounded using Wikipedia's medical articles. Articles are fetched, chunked, embedded (Transformers.js or OpenAI), and stored in Postgres for semantic retrieval.
RAG · Refreshed on each new question
🧬
Disease Registry
20 diseases defined in src/diseases/registry.ts. Imaging diseases use deterministic image scoring (no real CNN in demo mode). Clinical diseases use validated scoring rules (Framingham, CKD-EPI, NAFLD score).
Local · src/diseases/registry.ts
📄
PDF Report Extraction
Uploaded PDFs are parsed locally using pdf-parse. Text is matched against disease keyword dictionaries. No PHI leaves the server.
Local · pdf-parse library
👨‍⚕️
Synthetic Care Plans
All doctors, hospitals, medications, and care plans are completely synthetic and fictional. Generated deterministically from a date-seeded RNG so they refresh daily. Defined in src/care/dataPools.ts.
Synthetic · daily refresh
🗄
Postgres / Neon
All Q&A history, RAG embeddings, and healer fix-pattern cache are stored in Postgres (Neon for production). The answer history page reads directly from /api/answers.
Postgres · Neon (free tier prod)
🤖
LLM Providers
Patient chat synthesis uses whichever LLM API key you set: Anthropic Claude, Google Gemini (free), Groq/Llama (free), or OpenAI. The active provider is shown in the chat header.
Anthropic · Gemini · Groq · OpenAI
🔧
Auto-Healer (LLM-powered)
Runtime errors are parsed, sent to an LLM with file context, and a search/replace patch is generated and applied. Fix patterns are cached in Postgres. Source: src/healer/.
LLM · Postgres cache
🌐
MCP Tools (FHIR)
When MCP transport is active, the app exposes FHIR care-gap analysis tools using a synthetic FHIR adapter. Patient data is synthetic; no real EHR connection.
MCP · Synthetic FHIR adapter
